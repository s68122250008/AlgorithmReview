## กรณีศึกษาที่ 1: Array — คะแนนสอบนักศึกษา

**ไฟล์:** `src/ArrayCaseStudy.java`

โจทย์: มีคะแนนสอบของนักศึกษา 10 คนเก็บอยู่ใน `int[] scores` โปรแกรมคำนวณและแสดงผล:

1. คะแนนรวมของนักศึกษาทั้งหมด
2. คะแนนเฉลี่ย
3. คะแนนสูงสุด
4. คะแนนต่ำสุด
5. จำนวนนักศึกษาที่ได้คะแนนตั้งแต่ 7 ขึ้นไป
6. รายชื่อ (ลำดับที่) ของนักศึกษาที่ควรได้รับการทบทวนเพิ่มเติม (คะแนนต่ำกว่า 5)

**ข้อมูลตัวอย่าง:**
```
scores = {6, 8, 4, 9, 7, 5, 10, 3, 8, 2}
```

**ตัวอย่างผลลัพธ์:**
```
1. คะแนนรวม = 62.0
2. คะแนนเฉลี่ย = 6.2
3. คะแนนสูงสุด = 10
4. คะแนนต่ำสุด = 2
5. จำนวนนักศึกษาที่ได้ >= 7 คะแนน = 5
6. นักศึกษาที่ควรทบทวน: คนที่ 3 ได้ 4 คะแนน คนที่ 8 ได้ 3 คะแนน คนที่ 10 ได้ 2 คะแนน
```

## กรณีศึกษาที่ 2: Stack — ระบบ Undo ในโปรแกรมจดบันทึก

**ไฟล์:** `src/StackCaseStudy.java`

โจทย์: จำลองระบบ Undo โดยใช้ `Stack<String>` ซึ่งทำงานตามหลักการ **LIFO (Last In, First Out)** — คำสั่งล่าสุดที่ผู้ใช้ทำจะถูกยกเลิกก่อนเสมอ

**ลำดับคำสั่งที่ผู้ใช้ทำ (push เข้า Stack ตามลำดับ):**
```
Type Data → Type Structure → Delete Structure → Type Algorithm → Type Java
```

**การทำงานของโปรแกรม:**
1. เพิ่มคำสั่งทั้งหมดลงใน Stack ด้วย `push()`
2. แสดงสถานะของ Stack ทั้งหมด (บนสุด → ล่างสุด)
3. Undo คำสั่งล่าสุด 2 ครั้งด้วย `pop()` โดยตรวจสอบ `isEmpty()` ก่อนทุกครั้งเพื่อป้องกัน `EmptyStackException`
4. แสดงคำสั่งที่ถูก Undo ในแต่ละครั้ง
5. แสดงสถานะของ Stack หลังจาก Undo

**หลักการ LIFO ในกรณีนี้:** คำสั่ง `"Type Java"` ถูกพิมพ์ล่าสุด (เข้า Stack ทีหลังสุด) จึงเป็นคำสั่งแรกที่ถูก Undo (ออกจาก Stack ก่อน) ตรงข้ามกับ Queue ที่จะเรียงตามลำดับการเข้าก่อน-หลัง

**ตัวอย่างผลลัพธ์:**
```
สถานะ Stack 
บนสุด Type Java
        Type Algorithm
        Delete Structure
        Type Structure
        Type Data
Undo 2 ครั้ง
Undo ครั้งที่ 1: ยกเลิกคำสั่ง Type Java
Undo ครั้งที่ 2: ยกเลิกคำสั่ง Type Algorithm
สถานะ Stack หลังจาก Undo
บนสุด Delete Structure
        Type Structure
        Type Data
```

## กรณีศึกษาที่ 3: Queue — ระบบคิวผู้ป่วยในคลินิกสุขภาพ

**ไฟล์:** `src/QueueCaseStudy.java`

โจทย์: จำลองระบบคิวผู้ป่วยโดยใช้ `Queue<String>` (implement ด้วย `LinkedList`) ซึ่งทำงานตามหลักการ **FIFO (First In, First Out)** — ผู้ป่วยที่มาลงทะเบียนก่อนจะได้รับบริการก่อนเสมอ เพื่อความเป็นธรรม

**ลำดับผู้ป่วยที่มาลงทะเบียน:**
```
P001, P002, P003, P004, P005
```

**เหตุการณ์ที่เกิดขึ้น:**
1. เพิ่มผู้ป่วย P001–P005 เข้า Queue ด้วย `add()`
2. แสดงสถานะของ Queue เริ่มต้นทั้งหมด (หน้าแถว → ท้ายแถว)
3. เรียกผู้ป่วยออกจากคิว 2 คนด้วย `remove()` (P001 และ P002 ได้รับบริการแล้ว) โดยตรวจสอบ `isEmpty()` ก่อนทุกครั้งเพื่อป้องกัน `NoSuchElementException`
4. มีผู้ป่วยใหม่ P006 และ P007 มาลงทะเบียนเพิ่ม (ต่อท้าย Queue ด้วย `add()`)
5. แสดงผู้ป่วยคนถัดไปด้วย `peek()` (ดูอย่างเดียว ไม่เอาออกจากคิว)
6. แสดงจำนวนผู้ป่วยที่ยังรออยู่ด้วย `size()`
7. แสดงสถานะของ Queue ทั้งหมดหลังจากดำเนินการ

**หลักการ FIFO ในกรณีนี้:** P001 ลงทะเบียนก่อนใครเพื่อน (เข้า Queue ก่อนสุด) จึงเป็นผู้ป่วยคนแรกที่ถูกเรียกเข้ารับบริการ (ออกจาก Queue ก่อนสุด) ตามด้วย P002 แม้ P006 และ P007 จะมาลงทะเบียนทีหลัง P003–P005 แต่ก็ต้องต่อท้ายแถวและรอคิวตามลำดับ ไม่สามารถแซงคิวผู้ป่วยที่มาก่อนได้ สะท้อนความเป็นธรรมของระบบคิว

**ตัวอย่างผลลัพธ์:**
```
สถานะ Queue เริ่มต้น
หน้าแถวP001 ลำดับที่ 1
          P002 ลำดับที่ 2
          P003 ลำดับที่ 3
          P004 ลำดับที่ 4
          P005 ลำดับที่ 5
เรียกผู้ป่วยเข้ารับบริการ 2 คน
เรียกคนที่ 1: P001 เข้ารับบริการ
เรียกคนที่ 2: P002 เข้ารับบริการ
มีผู้ป่วยใหม่มาลงทะเบียนเพิ่ม
เพิ่ม P006 และ P007 เข้า Queue แล้ว
ตรวจสอบผู้ป่วยคนถัดไป
ผู้ป่วยคนถัดไปที่จะได้รับบริการคือ: P003
มีผู้ป่วยรออยู่ทั้งหมด 5 คน
สถานะ Queue สุดท้าย
หน้าแถวP003 ลำดับที่ 1
          P004 ลำดับที่ 2
          P005 ลำดับที่ 3
          P006 ลำดับที่ 4
          P007 ลำดับที่ 5
```

## สรุปเปรียบเทียบ Stack vs Queue

| | Stack (LIFO) | Queue (FIFO) |
|---|---|---|
| หลักการ | เข้าหลังสุด ออกก่อน | เข้าก่อนสุด ออกก่อน |
| เพิ่มข้อมูล | `push()` | `add()` / `offer()` |
| เอาข้อมูลออก | `pop()` | `remove()` / `poll()` |
| ดูข้อมูลโดยไม่เอาออก | `peek()` | `peek()` |
| ตัวอย่างการใช้งานจริง | ระบบ Undo, ปุ่ม Back ของเบราว์เซอร์ | คิวผู้ป่วย, คิวพิมพ์เอกสาร |
